
## 6. 风险点 / 偏差记录

### 6.1 chmod pattern 末尾 `\b` 省略

PRD 给的模式末尾是 `\bchmod\s+(-R\s+)?777\s+/\b`。这个 `\b` 在字符串末尾（`/` 之后）
**无法匹配** —— `/` 是非 word 字符，下一个字符是字符串尾（非 word），没有 word/non-word 转换，
所以 `\b` 在那里判定为 False，导致 `chmod 777 /` 漏报。

**修复**：改为 `\bchmod\s+(-R\s+)?777\s+/`，去掉末尾 `\b`。这同时放行了 `chmod 777 /etc/passwd`
等"以 / 开头的任意路径"——符合"777 触 root 子树"的实际风险语义，反而更安全。

### 6.2 大小写策略

所有 shell 模式按 `re.IGNORECASE` 编译。理由：`RM`、`RMDIR`、`format C:` 等大小写变体在
真实命令中很常见（PowerShell / Git Bash / WSL 都接受），严格小写匹配会产生大量漏报。
副作用：可能误拦 `GRM` 类含 rm 子串的工具，但当前项目无此类依赖。

### 6.3 默认行为：抛 vs 阻断

`check_shell` / `check_operation` 只抛异常，**不直接阻断**。这是设计决策：
- PRD §6.3 把"黑名单"定义为"需要用户确认的操作"，不是"绝对禁止"
- 真正的 prompt / bypass 流程应该由上层（如 tool dispatcher、shell runner）实现
- 本切片只交付"检测 + 抛错"原语；未来切片（S5.4 prompt 提示 / S5.5 bypass 审计）会接力
