---

## 1. 切片方法论

### 1.1 切片必须满足 5 条

| # | 条件 | 含义 |
|---|------|------|
| K1 | **可独立验收** | 完成后能用一条命令验证 |
| K2 | **可独立交付** | 不阻塞下个切片；可合入主干 |
| K3 | **可独立回滚** | 出问题 revert 一个 commit 就行 |
| K4 | **有明确边界** | 不超过 1 个 agent / 1 个模块 |
| K5 | **有验收脚本** | 跑一个测试或一条命令就能验 |

### 1.2 切片命名

```
S{M}.{N}  v0.1  短描述

例：S1.1 v0.1 项目骨架与配置入口
```

### 1.3 切片流程

```
1. 拉分支 slice/S<n>.<m>
2. 写测试 / 验收脚本（红）
3. 实现（绿）
4. 重构
5. 跑 lint + type check + test
6. 跑验收命令
7. 用户验收
8. commit + push
9. 更新 CHANGELOG.md
```

### 1.4 估时

- 平均 1-3 天/切片
- 总 48 切片 / 2-3 个月（单人）
- 超 5 天必须拆分

## 6. 切片开工仪式

```
1. 拉分支 slice/S<n>.<m>
2. 在 CHANGELOG.md 写"今天做什么、验收标准"
3. 写测试 / 验收脚本（红）
4. 实现（绿）
5. 跑完整链路（无回归）
6. 自检 DESIGN.md（如果是 UI 改动）
7. 跑"反 AI 味清单"（UI 改动）
8. 让用户试一次
9. 合入主干
10. 更新 CHANGELOG.md 标 done
```

## 7. 风险与回退

| 风险 | 缓解 |
|------|------|
| 切片估时严重不准 | 超 2 倍重新评估后续切片 |
| 单个切片卡住超 1 周 | 跳过，回头再补 |
| 设计风格被推翻 | DESIGN.md 是宪法，改它要重新评估受影响切片 |
| LLM provider 频繁变化 | 抽象层保住不变（不重写切片） |
| Obsidian vault 格式变化 | 解析层独立模块，不影响其他 |
